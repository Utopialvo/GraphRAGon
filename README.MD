# GraphRAGon с Graph of Thoughts

Это система, которая читает текст, строит по нему граф знаний (сущности, связи, контекст) и отвечает на вопросы, используя этот граф и локальную языковую модель. Главная фишка — вопрос обрабатывается цепочкой мыслей (Graph of Thoughts): генерируется несколько вариантов ответа с разных точек зрения, они улучшаются, оцениваются и сливаются в один связный результат.

Проект не претендует на production-grade, но внутри довольно аккуратный: кеширование, транзакции на загрузке, обработка конфликтов между одинаковыми фактами, потокобезопасная работа с графом. Сейчас весь GoT собран на LangGraph, а модель эмбеддингов и LLM подключаются напрямую через ленгчейновские классы — стало меньше самописных обёрток.

## Что делает

1. **Загрузка текста**  
   Большой текст режется на куски (чанки) с перекрытием. Для каждого куска LLM извлекает сущности и связи между ними.  
   *Пример:* в тексте «Иван нарубил дрова, Варвара растопила печь» появятся сущности `Иван` (PERSON), `дрова` (OBJECT), `печь` (OBJECT) и отношения `(Иван, рубил, дрова)`, `(Варвара, растопила, печь)`. Всё это пишется в Memgraph вместе с исходным текстом чанков. Каждый чанк обрабатывается в отдельной транзакции — если внутри чанка что-то пошло не так, транзакция откатывается, но остальные чанки продолжают загружаться.

2. **Хранилище Memgraph**  
   Сущности, отношения и пассажи хранятся в графовой базе. Для быстрого поиска по смыслу используется векторный индекс (эмбеддинги генерируются локальной sentence‑transformer моделью). Все обращения к графу защищены блокировками, поэтому конкурентные запросы не поломают данные. Массовая загрузка текста дополнительно оборачивает каждый чанк в транзакцию.

3. **Разрешение конфликтов (опционально)**  
   Когда из разных кусков текста для одних и тех же сущностей появляются несовместимые отношения (например, «Иван купил молоко» и «Иван продал молоко»), включается LLM‑арбитр. Он решает, какое отношение оставить, или предлагает объединённый вариант. Конфликты разруливаются итеративно для каждой пары сущностей. Этот шаг не выполняется автоматически при загрузке текста — его нужно запустить явно (метод `graph.resolve_conflicts()`).

4. **Ответы на вопросы**  
   Основной метод `ask()` может работать в двух режимах.

   **Без Graph of Thoughts** (`use_got=False`)  
   Контекст собирается по векторному поиску ближайших сущностей, к ним подтягиваются связанные отношения и куски текста (плюс небольшой случайный обход соседей для охвата). Дальше этот контекст и вопрос отправляются в LLM, ответ возвращается как есть.

   **С Graph of Thoughts (по умолчанию)**  
   Вместо линейного пайплайна строится граф состояний (StateGraph из LangGraph), который управляет жизнью «мыслей» (текстовых ответов-рассуждений). Узлы графа — это операции, а рёбра задают переходы. Стандартный граф выглядит так:

   - *Генерация (Generate)* — создаётся `got_num_thoughts` (по умолчанию 3) мыслей с разными ролями (Аналитик, Критик, Стратег, Скептик, Учёный). Роли выбираются случайно (могут повторяться, если мыслей больше, чем ролей).
   - *Оценка (Score)* — каждая мысль получает числовую оценку от 0 до 10.
   - *Селекция (Select)* — опционально оставляются только `got_select_top_k` лучших (управляется флагом `got_use_select`).
   - *Улучшение (Refine)* — каждая отобранная мысль улучшается (становится точнее, полнее, логичнее). Этот шаг может повторяться: параметр `got_iterations` задаёт, сколько раз мы пройдём цикл refine → score → select. По умолчанию 1, то есть одно улучшение.
   - *Слияние (Merge)* — все итоговые мысли объединяются в единый связный ответ.

   Температуры для каждой операции настраиваются отдельно. Можно полностью отключить оценку, селекцию или слияние через конфиг.  

   Результат кешируется (на уровне LLM-запросов), а история вопросов‑ответов хранится в оперативной памяти.

## Структура проекта

Теперь весь GoT упакован в один файл `got_graph.py`, а LLM и эмбеддинги подключаются через ленгчейн. Старые обёртки `llm_client.py`, `embedding_client.py` и `utils.py` ушли, пакет `got/` расформирован.

```
.
├── config.yaml              # Конфигурация: пути к LLM, Memgraph, параметры чанков, GoT и т.д.
├── docker-compose.yml       # Быстрый подъём Memgraph + Memgraph Lab
├── src/
│   ├── graph_rag.py         # Точка входа: класс GraphRAG, сборка всех компонентов
│   ├── memgraph_store.py    # Всё для Memgraph: сущности, отношения, пассажи, поиск
│   ├── conflict_resolver.py # Разруливание противоречивых отношений
│   ├── llm_entity_extract.py# Извлечение сущностей LLM из куска текста
│   ├── relation_extractor.py# Извлечение отношений между сущностями
│   ├── history_manager.py   # Простая история диалогов в оперативке
│   ├── got_graph.py         # Graph of Thoughts на LangGraph
│   └── __init__.py          
├── Test.ipynb               # Пример использования в Jupyter
└── README.md
```

## Железо и зависимости

- **Memgraph** (запускается через `docker-compose up` или отдельно).  
- **Локальная LLM** с OpenAI-совместимым API (например, vllm, text-generation-webui, LM Studio, llama.cpp с сервером). Мы тестировали на Qwen 3.5-9B-Q4_0.  
- **Python 3.10+** с пакетами:  
  ```bash
  pip install pyyaml pydantic langchain langchain-openai langchain-huggingface langchain-community langgraph neo4j sentence-transformers
  ```
- Модель эмбеддингов по умолчанию `all-MiniLM-L6-v2`, она скачается при первом запуске сама.  
- Кэш LLM пишется в `llm_cache.db` (SQLite). Можно задать другой путь в конфиге.

## Быстрый старт

1. Поднимите Memgraph (рекомендуем через наш docker-compose, он добавит ещё и веб-интерфейс Lab):  
   ```bash
   docker network create internal-network  # если ещё нет
   docker-compose up -d
   ```
   Или просто: `docker run -p 7687:7687 memgraph/memgraph`.

2. Запустите вашу LLM, убедитесь, что она отвечает на эндпоинте `http://your-llm:8080/v1/chat/completions`.

3. Создайте `config.yaml` со своими параметрами (пример ниже).

4. Запустите Jupyter или обычный скрипт:
   ```python
   from graph_rag import GraphRAG, GraphRAGConfig

   config = GraphRAGConfig.from_yaml("config.yaml")
   graph = GraphRAG(config)

   # Загрузим текст (возвращает список ID созданных чанков)
   ids = graph.process_text("Иван нарубил дрова. Варвара растопила печь. ...")

   # Опционально: разрешить конфликты между отношениями
   # conflicts = graph.resolve_conflicts()

   # Зададим вопрос с GoT (по умолчанию)
   answer = graph.ask("Что делала Варвара?")
   print(answer)

   # Или без GoT
   answer_direct = graph.ask("Что делал Иван?", use_got=False)
   print(answer_direct)
   ```

5. При завершении скрипта (или при вызове `graph.close()`) будет создан снапшот Memgraph, чтобы граф не потерялся.

## Нюансы и ограничения

- **Параллелизм** — Graph of Thoughts выполняется последовательно: операции идут одна за другой, параллельных запросов к LLM в рамках одного вопроса нет. Однако хранилище (MemgraphStore) потокобезопасно, так что несколько потоков могут одновременно дёргать `ask()` или `process_text()` без риска.
- **Конфликты** — не разрешаются автоматически при загрузке текста. Если нужна консистентность отношений, вызывайте `graph.resolve_conflicts()` руками.
- **Кеш** — LLM‑запросы кешируются по полному промпту (включая температуру). Старые записи чистятся автоматически по таймауту.
- **Векторный индекс** — создаётся при первом подключении, размерность определяется моделью. При смене модели индекс пересоздаётся.
- **Загрузка текста** — каждый чанк в отдельной транзакции: ошибка в одном чанке не рушит остальные.
- **Очистка графа** — `graph.clear()` удаляет все узлы и связи, сбрасывает метрики и историю, пересоздаёт пустой индекс.
- **Роли в GoT** — роли выбираются случайно из списка, повторы возможны, если мыслей больше, чем уникальных ролей. Хотите строгое чередование — выключите `got_use_random_roles`.
- **Гибкость графа** — благодаря LangGraph вы можете менять логику переходов прямо в `got_graph.py` без перелопачивания всей системы.
- **Граф общий** — нет встроенного разделения сессий или пользователей. Все данные живут в одном графе.

## Чего здесь нет

- UI / веб‑интерфейса — только Python API.
- Поддержки нескольких пользователей / сессий.
- Автоматического разрешения конфликтов в процессе загрузки.
- Продвинутой обработки ошибок LLM (если модель глючит, вы получите пустой ответ или fallback).

## Зачем это всё

Проект делался как proof‑of‑concept: можно ли на коленке собрать RAG, который не просто ищет куски текста, а пытается «понимать» факты, рассуждать с разных точек зрения и выдавать осмысленный ответ. На небольших текстах работает хорошо, на больших — упирается в скорость LLM и память графа. Приятный бонус — почти всё кэшируется, поэтому повторные запуски на тех же данных летают.

Если захотите доработать под свои задачи — посмотрите методы `process_text` и `ask`, оттуда всё расходится по компонентам. Вопросы, баги и предложения принимаются в Issues.
