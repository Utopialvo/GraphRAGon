# GraphRAGon с Graph of Thoughts

Это система, которая читает текст, строит по нему граф знаний (сущности, связи, контекст) и отвечает на вопросы, используя этот граф и локальную языковую модель. Главная фишка — вопрос обрабатывается цепочкой мыслей (Graph of Thoughts): генерируется несколько вариантов ответа с разных точек зрения, они улучшаются, оцениваются и сливаются в один связный результат.

Проект не претендует на production-grade, но внутри довольно аккуратный: кеширование, транзакции на загрузке, обработка конфликтов между одинаковыми фактами, потокобезопасная работа с графом.

## Что делает

1. **Загрузка текста**  
   Большой текст режется на куски (чанки) с перекрытием. Для каждого куска LLM извлекает сущности и связи между ними.  
   *Пример:* в тексте «Иван нарубил дрова, Варвара растопила печь» появятся сущности `Иван` (PERSON), `дрова` (OBJECT), `печь` (OBJECT) и отношения `(Иван, рубил, дрова)`, `(Варвара, растопила, печь)`. Всё это пишется в Memgraph вместе с исходным текстом чанков. Каждый чанк обрабатывается в отдельной транзакции — если внутри чанка что-то пошло не так, транзакция откатывается, но остальные чанки продолжают загружаться.

2. **Хранилище Memgraph**  
   Сущности, отношения и пассажи хранятся в графовой базе. Для быстрого поиска по смыслу используется векторный индекс (эмбеддинги генерируются локальной sentence‑transformer моделью). Все обращения к графу защищены блокировками, поэтому конкурентные запросы не поломают данные. Массовая загрузка текста дополнительно оборачивает каждый чанк в транзакцию.

3. **Разрешение конфликтов (опционально)**  
   Когда из разных кусков текста для одних и тех же сущностей появляются несовместимые отношения (например, «Иван купил молоко» и «Иван продал молоко»), включается LLM‑арбитр. Он решает, какое отношение оставить, или предлагает объединённый вариант. Конфликты разруливаются итеративно для каждой пары сущностей. Этот шаг не выполняется автоматически при загрузке текста — его нужно запустить явно (метод `graph.resolve_conflicts()`).

4. **Ответы на вопросы**  
   Основной метод `ask()` может работать в двух режимах.

   **Без Graph of Thoughts** (`use_got=False`)  
   Контекст собирается по векторному поиску ближайших сущностей, к ним подтягиваются связанные отношения и куски текста (плюс небольшой случайный обход соседей для охвата). Дальше этот контекст и вопрос отправляются в LLM, ответ возвращается как есть.

   **С Graph of Thoughts (по умолчанию)**  
   Вместо линейного пайплайна строится **граф операций (Graph of Operations)**, который позволяет гибко комбинировать этапы рассуждения. Узлы графа — это операции над «мыслями» (текстовыми ответами-рассуждениями), а рёбра задают зависимости между ними. Стандартный граф включает:

   - *Генерация (Generate)* — создаётся `got_num_thoughts` (по умолчанию 3) мыслей с разными ролями (Аналитик, Критик, Стратег, Скептик, Учёный), что обеспечивает многогранность.
   - *Оценка (Score)* — каждая мысль получает числовую оценку от 0 до 10.
   - *Селекция (Select)* — опционально оставляются только `got_select_top_k` мыслей с наивысшей оценкой (включается параметром `got_use_select`).
   - *Улучшение (Refine)* — каждая отобранная мысль улучшается (становится точнее, полнее, логичнее). Количество циклов улучшения задаётся параметром `got_iterations` (по умолчанию 1). На каждом цикле можно повторно запускать оценку.
   - *Слияние (Merge)* — все итоговые мысли объединяются в единый связный ответ.

   Граф выполняется в топологическом порядке, что позволяет легко изменять структуру (например, добавить ветвления или дополнительные итерации) без правки основного кода. Температуры для каждой операции настраиваются отдельно. Роли можно задать списком или выбирать случайно (без повторений в рамках одной генерации).

   Результат кешируется (на уровне LLM-запросов), а история вопросов‑ответов хранится в оперативной памяти.

## Железо и зависимости

- **Memgraph** (должен быть запущен, порт и хост в конфиге).  
- **Локальная LLM** с OpenAI-совместимым API (например, vllm, text-generation-webui или LM Studio с эндпоинтом `/v1`). Проект заточен под модели вроде Qwen‑9B‑GGUF, но подойдёт любая.  
- **Python 3.10+** с пакетами:  
  ```bash
  pip install pyyaml pydantic openai gqlalchemy sentence-transformers
  ```
- Модель эмбеддингов по умолчанию `all-MiniLM-L6-v2`, она скачается при первом запуске сама.  
- Кэш LLM и эмбеддингов пишется в `llm_cache.db` (SQLite). Можно задать другой путь в конфиге.

## Быстрый старт

1. Поднимите Memgraph (например, Docker):  
   `docker run -p 7687:7687 memgraph/memgraph`

2. Запустите вашу LLM, убедитесь, что она отвечает на эндпоинте `http://your-llm:8080/v1/chat/completions`.

3. Создайте `config.yaml` со своими параметрами (пример ниже).

4. Запустите Jupyter или обычный скрипт:
   ```python
   from graph_rag import GraphRAG, GraphRAGConfig

   config = GraphRAGConfig.from_yaml("config.yaml")
   graph = GraphRAG(config)

   # Загрузим текст (возвращает список ID созданных чанков)
   ids = graph.process_text("Иван нарубил дрова. Варвара растопила печь. ...")

   # Опционально: разрешить конфликты между отношениями
   # conflicts = graph.resolve_conflicts()

   # Зададим вопрос с GoT (по умолчанию)
   answer = graph.ask("Что делала Варвара?")
   print(answer)

   # Или без GoT
   answer_direct = graph.ask("Что делал Иван?", use_got=False)
   print(answer_direct)
   ```

5. При завершении скрипта (или при вызове `graph.close()`) будет создан снапшот Memgraph, чтобы граф не потерялся.

## Нюансы и ограничения

- **Параллелизм** — Graph of Thoughts выполняется последовательно: граф операций обходится в топологическом порядке, операции выполняются одна за другой. Параллельных запросов к LLM в рамках одного вопроса нет. Однако сам графовый движок (MemgraphStore) потокобезопасен благодаря блокировкам, поэтому несколько потоков могут одновременно дёргать `ask()` или `process_text()` без риска сломать данные. Гибкость графа не приводит к дополнительным накладным расходам на синхронизацию.
- **Конфликты** — не разрешаются автоматически при загрузке текста. Если для вашего сценария важна консистентность отношений, нужно явно вызвать `graph.resolve_conflicts()`. Метод работает аккуратно, итеративно для каждой пары сущностей. При неудаче арбитража конфликт логируется, граф не ломается.
- **Кеш** — LLM‑запросы кешируются по полному промпту, включая температуру. При отладке это сильно ускоряет работу. Старые записи чистятся автоматически (настраивается).
- **Векторный индекс** — создаётся при первом подключении к Memgraph (если его ещё нет). Размерность автоматически берётся из модели эмбеддингов. При смене модели индекс пересоздаётся. При очистке графа через `clear()` индекс гарантированно восстанавливается, даже если удаление старого упало.
- **Загрузка текста** — каждый чанк обрабатывается в отдельной транзакции. Если внутри чанка возникает ошибка, транзакция откатывается, и этот чанк пропускается, остальные продолжают загружаться. Это позволяет не терять весь прогресс из-за одного сбойного куска.
- **Очистка графа** — `graph.clear()` удаляет все узлы и связи, сбрасывает метрики и историю, после чего пересоздаёт пустой векторный индекс. Удобно для экспериментов.
- **Роли в GoT** — по умолчанию роли для генерации мыслей выбираются случайно без повторов. Если ролей меньше, чем число мыслей, оставшиеся добираются из полного списка (могут повторяться). Поведение можно изменить в коде операции `Generate`.
- **Гибкость Graph of Operations** — в отличие от жёсткого пайплайна, граф операций позволяет легко изменять порядок и состав этапов. Например, можно убрать `Select`, добавить несколько параллельных веток `Generate` или повторить `Refine` несколько раз подряд. Это делает систему адаптируемой под разные типы вопросов без изменения кода — достаточно настроить параметры или модифицировать сборку графа в методе `ask()`.
- **Граф общий** — нет встроенного разделения сессий или пользователей. Все данные живут в одном графе, так что при одновременной работе нескольких экземпляров учитывайте, что они делят одно хранилище.

## Чего здесь нет

- UI / веб‑интерфейса — только Python API.
- Поддержки нескольких пользователей / сессий (граф общий).
- Продвинутой обработки ошибок LLM (если модель глючит, вы получите пустой ответ или fallback).
- Автоматического разрешения конфликтов в процессе загрузки текста.

## Зачем это всё

Проект делался как proof‑of‑concept: можно ли на коленке собрать RAG, который не просто ищет куски текста, а пытается «понимать» факты, рассуждать с разных точек зрения и выдавать осмысленный ответ. На небольших текстах работает хорошо, на больших — упирается в скорость LLM и память графа. Приятный бонус — почти всё кэшируется, поэтому повторные запуски на тех же данных летают.

Если захотите доработать под свои задачи — посмотрите методы `process_text` и `ask`, оттуда всё расходится по компонентам. Вопросы, баги и предложения принимаются в Issues.
