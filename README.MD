# GraphRAGon с Graph of Thoughts

Это система, которая читает текст, строит по нему граф знаний (сущности, связи, контекст) и отвечает на вопросы, используя этот граф и локальную языковую модель. Главная фишка — вопрос может разбиваться на подвопросы (Graph of Thoughts), ответы на них собираются вместе, а результат звучит связно, а не как мешанина.

Проект не претендует на production-grade, но внутри довольно аккуратный: кеширование, транзакции, обработка конфликтов между одинаковыми фактами, параллельная работа с графом.

## Что делает

1. **Загрузка текста**  
   Делит большой текст на куски (чанки) и по каждому запускает LLM для извлечения сущностей и отношений.  
   *Пример:* в тексте «Иван нарубил дрова, Варвара растопила печь» появятся сущности `Иван` (PERSON), `дрова` (OBJECT), `печь` (OBJECT) и отношения `(Иван, рубил, дрова)`, `(Варвара, растопила, печь)`.

2. **Хранилище Memgraph**  
   Все сущности, отношения и текстовые куски сохраняются в графовую базу Memgraph. Для быстрого поиска по смыслу используется векторный индекс.

3. **Разрешение конфликтов**  
   Когда из разных частей текста для одних и тех же сущностей извлекаются несовместимые отношения (например, «Иван купил молоко» и «Иван продал молоко»), система запускает LLM‑арбитра, который решает, какое отношение оставить, или объединяет их в одно.

4. **Ответы на вопросы**  
   Вопрос может проходить через Graph of Thoughts:
   - LLM делит сложный вопрос на простые (например, «Что делал Иван утром и вечером?» → «Что Иван делал утром?», «Что Иван делал вечером?»).
   - Каждый подвопрос обрабатывается отдельно, контекст подбирается из графа (плюс небольшой случайный обход соседей для лучшего охвата).
   - Ответы сливаются в один итоговый текст.

   Можно отключить GoT и задавать вопросы «в лоб» — тогда контекст возьмётся по векторному поиску сущностей, но без декомпозиции.

## Железо и зависимости

- **Memgraph** (должен быть запущен, порт и хост в конфиге).  
- **Локальная LLM** с OpenAI-совместимым API (например, vllm или text-generation-webui с эндпоинтом `/v1`). Проект заточен под модели вроде Qwen‑9B‑GGUF, но подойдёт любая.  
- **Python 3.10+** с пакетами:  
  `pip install pyyaml pydantic openai gqlalchemy sentence-transformers`

Модель эмбеддингов по умолчанию `all-MiniLM-L6-v2`, она скачается при первом запуске сама.  
Кэш LLM и эмбеддингов пишется в `llm_cache.db` (SQLite). Можно задать другой путь в конфиге.

## Быстрый старт

1. Поднимите Memgraph (например, Docker):  
   `docker run -p 7687:7687 memgraph/memgraph`

2. Запустите вашу LLM, убедитесь, что она отвечает на эндпоинте `http://your-llm:8080/v1/chat/completions`.

3. Создайте `config.yaml` со своими параметрами (пример ниже).

4. Запустите Jupyter или обычный скрипт:
   ```python
   from graph_rag import GraphRAG, GraphRAGConfig

   config = GraphRAGConfig.from_yaml("config.yaml")
   graph = GraphRAG(config)

   # Загрузим текст
   graph.process_text("Иван нарубил дрова. Варвара растопила печь. ...")

   # Зададим вопрос
   answer = graph.ask("Что делала Варвара?", use_got=True)
   print(answer)
   ```

5. При завершении скрипта (или если вызвать `graph.close()`) будет создан снапшот Memgraph, чтобы граф не потерялся.

## Нюансы и ограничения

- **Параллелизм** — если включён `got_parallel`, подвопросы обрабатываются в нескольких потоках. Для каждого потока создаётся свой LLM‑клиент, чтобы не было гонок в HTTP‑соединении. Блокировки в Memgraph тоже учтены, так что краха не будет, но при большом количестве потоков может возрасти потребление памяти.
- **Конфликты** — решаются аккуратно, итеративно для каждой пары сущностей. Если один и тот же факт встречается много раз, LLM может не справиться, но базовые противоречия разруливает хорошо.
- **Кеш** — LLM‑запросы кешируются по полному промпту. При отладке это сильно ускоряет работу. Старые записи чистятся автоматически.
- **Векторный индекс** — создаётся при первом добавлении сущности (размерность подбирается под модель эмбеддингов). Если модель меняется, индекс автоматически пересоздастся.
- **Очистка графа** — `graph.clear()` удаляет вообще всё, включая векторный индекс, после чего сразу создаёт его заново (пустым). Это удобно для экспериментов.

## Чего здесь нет

- UI / веб‑интерфейса — только Python API.
- Поддержки нескольких пользователей / сессий (граф общий).
- Продвинутой обработки ошибок LLM (если модель глючит, вы получите пустой ответ или fallback).

## Зачем это всё

Проект делался как proof‑of‑concept: можно ли на коленке собрать RAG, который не просто ищет куски текста, а пытается «понимать» факты и выдавать осмысленный ответ. На небольших текстах работает хорошо, на больших — упирается в скорость LLM и память графа. Приятный бонус — почти всё кэшируется, поэтому повторные запуски на тех же данных летают.

Если захотите доработать под свои задачи — посмотрите методы `process_text` и `ask`, оттуда всё расходится по компонентам. Вопросы, баги и предложения принимаются в Issues.
